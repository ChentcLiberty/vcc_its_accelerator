# 题1 官方接口顶层第一版验证说明 v1

## 验证对象

- [its_top_official_if_stage1.v](../05_rtl/its_top_official_if_stage1.v)

## 这版做什么

这版不是再验证单个二维核，而是第一次验证“题面官方接口顶层”本身，包括：

1. `it_info` 解码
2. `it_data_in_req` 驱动
3. sparse 输入按 `it_data_addr` 恢复到内部块
4. `4x4` 路径的 `raster -> xBar` 映射
5. `4点/拍` 的 `40-bit` 打包输出
6. `it_done` 收尾

## 当前 case

1. `4x4 DCT2 + LFNST`
   - `tu_width = 4`
   - `tu_height = 4`
   - `tr_type_hor = DCT2`
   - `tr_type_ver = DCT2`
   - `lfnst_idx = 1`
   - `16` 个点按光栅地址送入官方顶层
   - 参考结果来自 [lfnst_idct2_2d4_core.v](../05_rtl/lfnst_idct2_2d4_core.v)
2. `8x8 DCT8`
   - `tu_width = 8`
   - `tu_height = 8`
   - `tr_type_hor = DCT8`
   - `tr_type_ver = DCT8`
   - `lfnst_idx = 0`
   - `64` 个点按光栅地址送入官方顶层
   - 参考结果来自 [its_2d8_core.v](../05_rtl/its_2d8_core.v)
3. `16x16 DCT8`
   - `tu_width = 16`
   - `tu_height = 16`
   - `tr_type_hor = DCT8`
   - `tr_type_ver = DCT8`
   - `lfnst_idx = 0`
   - `256` 个点按光栅地址送入官方顶层
   - 参考结果来自离线 golden 文件 [its_2d16_dct8_full_expected.memh](../06_tb/data/its_2d16_dct8_full_expected.memh)
4. unsupported 模式
   - 用 `16x8 DCT2` 输入确认顶层走 unsupported 分支
   - 检查 `it_done`
   - 检查不产生 `it_data_out_vld`
5. `it_data_out_req` 单拍反压
   - 插在 `16x16 DCT8` 的流式输出阶段
   - 检查 `req=0` 时顶层不对外报 `vld`

## 参考策略

这版 smoke testbench 没有全部手抄一套期望矩阵，而是混合使用两类参考：

- live reference：

- `4x4 LFNST + IDCT2`
- `8x8 DCT8`

- 离线 golden：

- `16x16 DCT8` 直接复用已有 [its_2d16_dct8_full_expected.memh](../06_tb/data/its_2d16_dct8_full_expected.memh)

其中 live reference 路径继续实例化现有已通过的子链路模块，离线 golden 路径则直接用已通过 directed testbench 产物做 `40-bit` 打包比对。

这样这版更像“接口适配验证”，而不是再次重复验证内核数学本身。

## 当前边界

这版 smoke 还不是完整赛题顶层回归，只覆盖：

- `4x4 DCT2 + LFNST`
- `8x8 DCT8`
- `16x16 DCT8`
- unsupported 模式
- 一次 `it_data_out_req` 单拍反压

还没有覆盖：

- `8x8 DCT2`
- `8x8 DST7`
- `16x16 DCT2`
- `16x16 DST7`
- 更长时间或多次 `it_data_out_req` 反压场景

这些后面应该继续逐项补齐。
