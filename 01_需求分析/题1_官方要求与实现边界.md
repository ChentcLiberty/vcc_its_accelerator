# 题1 官方要求与实现边界

更新时间：2026-05-18

## 官方题面硬要求

根据官方赛题页和本地截图 [赛题要求.png](../00_题面资料/赛题要求.png)，题1要求：

1. 实现 `VVC` 反变换模块设计
2. 支持 `DCT2`、`DCT8`、`DST7` 三种反变换类型
3. 支持 `LFNST` 反二次变换
4. 反二次变换路径为：`先 LFNST，再 DCT2 反变换`
5. 数据按 `TU` 内光栅顺序输入
6. 输入接口为 `一拍一个点`，且只输入非零点
7. 计算性能要求 `一拍计算四个点`，支持流水
8. 输出按 `TU` 内光栅顺序输出，`一拍4个点`
9. 使用 `Verilog` 实现
10. 目标工作主频 `500MHz`
11. 追求面积和功耗最优
12. 需要覆盖官方列出的全部块大小与模式组合

官方赛题页还注明：`本题于 2026-04-24 有更新，附件数据同步更新，需要重新下载。`

    - `nTrs = 48` 时，输入只有左上 `4x4` 非零，输出只有左上 `3` 个 `4x4` 块非零

## 当前建议的实现边界

不建议一开始就把所有模式一起硬上。推荐按四阶段推进：

### 阶段A：LFNST打通

- 支持 `nTrs = 16 / 48`
- 支持 `lfnstTrSetIdx`、`lfnst_idx`
- 建好本地机器可读矩阵表
- 建立 Python/NumPy 参考模型
- 做 RTL + 自检 testbench

### 阶段B：1D 变换核打通

- 分别完成：
  - `IDCT2`
  - `IDCT8`
  - `IDST7`
- 先做 `4/8/16/32/64` 长度可参数化 1D 核
- 先不追最优面积，先确保公式和位宽正确

### 阶段C：2D ITS 顶层打通

- 输入稀疏系数装载
- `列变换 -> 转置/中间buffer -> 行变换`
- `LFNST bypass / enable`
- 光栅顺序输出整理

### 阶段D：性能和PPA收敛

- 拉到 `4点/拍` 输出
- 做流水切分
- 识别乘法器、buffer、转置 RAM 热点
- 做面积和功耗优化

## 当前最重要的工程判断

- 比赛题最终要全模式支持，但第一周不需要追完整覆盖
- 第一周最关键的是先把 `LFNST` 和 `参考模型` 收稳
- 没有可靠参考模型，后面的 `IDCT/DCT8/DST7` 只会越做越乱
- 之前“只输入非零点”的理解还不够完整；从截图看，输入不仅有数据值，还有显式地址 `it_data_addr`

## 题面明确的接口定义

以下内容来自 `赛题要求.png`，优先级高于之前的工程假设：

- `it_info[21:0]`
  - `[6:0] tu_width`
  - `[13:7] tu_height`
  - `[15:14] tr_type_hor`
  - `[17:16] tr_type_ver`
  - `[19:18] lfnst_tr_set_idx`
  - `[21:20] lfnst_idx`
  - 其中 `lfnst_idx = 0` 表示不需要 `LFNST`
- `it_info_vld`
  - `info` 有效脉冲
  - 时序要求：先发 `info`，再发 `data` 和 `end`
- `it_data_in[15:0]`
  - 有符号输入数据
  - 只输入非零数据，零数据跳过
- `it_data_addr[11:0]`
  - 当前输入数据在 `TU` 内的光栅扫描位置
- `it_data_in_vld`
  - 输入数据有效脉冲
- `it_data_end`
  - `TU` 块输入完成脉冲
  - 要么与 `it_data_in_vld` 同拍生效，要么单独生效
- `it_data_in_req`
  - 输入数据请求
  - 题面明确要求输入侧支持按点反压
  - 只有 `it_data_in_req = 1` 时，输入侧才拉高 `it_data_in_vld`
- `it_data_out[39:0]`
  - 输出 `4` 个结果
  - `[9:0]` 第一个
  - `[19:10]` 第二个
  - `[29:20]` 第三个
  - `[39:30]` 第四个
- `it_data_out_vld`
  - 输出数据有效脉冲
- `it_data_out_req`
  - 高电平有效
  - 只有 `it_data_out_req = 1` 时才允许输出，否则 `it_data_out_vld` 不能拉高
- `it_done`
  - `TU` 计算和输出完成脉冲

## 当前接口假设与修正

以下内容是结合题面后的工程判断：

- 外部输入数据位宽：`16-bit signed`
- 外部输出总线位宽：`40-bit`
- 单点输出位宽：当前按题面表格理解为 `10-bit` 打包输出
- 这和前面 `LFNST` 路径里的 `Clip3(-32768, 32767, ...)` 存在表面不一致
- 当前最可能的解释是：
  - 内部变换结果先按更高位宽计算/裁剪
  - 对外 `it_data_out` 为截位或规范化后的 `10-bit x 4` 打包结果
  - 这一点后续必须通过官方附件或参考代码再确认
- 内部累加位宽：建议先按 `32~40 bit` 预算
- `LFNST enable` 不需要单独输入，可由 `lfnst_idx != 0` 推导

## 当前已确认的矩阵组织方式

- `LFNST`
  - 本地 `Low frequency non.docx` 已成功解析出 `16` 组场景
  - 其中 `nTrs = 16` 有 `8` 组，`nTrs = 48` 有 `8` 组
- `DCT2`
  - 附件提供的是 `64-point` 基表
  - 文档里以 `Col0to15`、`Col16to31` 两个 `64x16` 列块给出
  - 剩余 `32..63` 列可按附件里的镜像/奇偶符号规则补齐
  - 当前脚本已按附件重建出完整 `64x64` 基表
  - `4/8/16/32` 点 `DCT2` 可由该 `64x64` 基表按 `64 / nTbS` 的行步长抽取得到
- `DST7 / DCT8`
  - `4/8/16` 点矩阵在附件中直接给出
  - `32-point` 附件只给出了前 `16` 行
  - 当前脚本做法是：
    - 前 `16` 行直接使用附件值
    - 后 `16` 行先按赛题给出的三角基函数解析式补齐
  - 这意味着 `DST7/DCT8 32-point` 现在已经可驱动模型和后续 RTL 开发，但仍保留“后半矩阵进一步对齐官方完整来源”的待确认项

## 题面明确的支持块型

### DCT2

- `4x4, 4x8, 4x16, 4x32, 4x64`
- `8x4, 16x4, 32x4, 64x4`
- `8x8, 8x16, 8x32, 8x64`
- `16x8, 32x8, 64x8`
- `16x16, 16x32, 16x64`
- `32x16, 32x32, 32x64`
- `64x16, 64x32, 64x64`

### DCT8 / DST7

- `4x4, 4x8, 4x16, 4x32`
- `8x4, 16x4, 32x4`
- `8x8, 8x16, 8x32`
- `16x8, 32x8`
- `16x16, 16x32`
- `32x16, 32x32`

### LFNST

- `nTrs = 16`
  - `4x4, 4x8, 4x16, 4x32, 4x64, 8x4, 16x4, 32x4, 64x4`
- `nTrs = 48`
  - `8x8, 8x16, 8x32, 8x64, 16x8, 32x8, 64x8`
  - `16x16, 16x32, 16x64, 32x16, 32x32, 32x64, 64x16, 64x32, 64x64`

## 结论

当前最优开工顺序不是“完整 ITS 一口气做完”，而是：

1. `LFNST 参考模型`
2. `LFNST RTL`
3. `1D 反变换核`
4. `2D 顶层整合`
