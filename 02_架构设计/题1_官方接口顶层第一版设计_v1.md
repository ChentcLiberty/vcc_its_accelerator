# 题1 官方接口顶层第一版设计 v1

## 这版做什么

这版第一次把题面里的官方接口真正落成 RTL 顶层，而不是只停留在子模块级：

- `it_info[21:0]`
- `it_data_in / it_data_addr / it_data_in_vld / it_data_end`
- `it_data_in_req`
- `it_data_out[39:0] / it_data_out_vld / it_data_out_req`
- `it_done`

对应文件：

- 顶层： [its_top_official_if_stage1.v](../05_rtl/its_top_official_if_stage1.v)

## 这版支持的模式

第一版不追求“题面全覆盖”，而是先落一版真实可综合、可回归的官方接口顶层。当前支持：

1. `4x4`
   - `tr_type_hor = DCT2`
   - `tr_type_ver = DCT2`
   - `lfnst_idx = 0/1/2` 都允许
   - 走现有 [lfnst_idct2_2d4_core.v](../05_rtl/lfnst_idct2_2d4_core.v)
2. `8x8`
   - 仅支持平方块
   - `DCT2 / DST7 / DCT8`
   - 要求 `tr_type_hor == tr_type_ver`
   - 当前要求 `lfnst_idx = 0`
3. `16x16`
   - 仅支持平方块
   - `DCT2 / DST7 / DCT8`
   - 要求 `tr_type_hor == tr_type_ver`
   - 当前要求 `lfnst_idx = 0`

不支持的配置当前会走“无输出、直接 `it_done`”的收尾路径，避免顶层卡死。

## 顶层数据流

当前顶层数据流是：

`it_info -> sparse coeff load(value + addr) -> internal TU buffer -> selected transform core -> 10-bit x4 pack -> official output`

其中有一个关键点：

- `4x4 LFNST` 路径不能直接把光栅顺序输入给 `lfnst_idct2_2d4_core`
- 该链路需要的是 `xBar` 扫描顺序
- 所以顶层在 `4x4` 路径里额外做了一次 `raster -> LFNST scan` 映射

当前映射使用的是 `4x4` 对角扫描顺序：

`0, 4, 1, 8, 5, 2, 12, 9, 6, 3, 13, 10, 7, 14, 11, 15`

## 握手策略

### 输入侧

- `it_data_in_req = 1` 只在 `LOAD` 状态拉高
- 也就是：
  - 已收到 `it_info`
  - 顶层正在等待 sparse 输入和 `it_data_end`

### 输出侧

当前内部计算核保留原来的 block-level 语义：`out_req=0` 时，`out_valid` 仍可能保持高电平但数据稳定。

题面官方接口要求更严格：

- 只有 `it_data_out_req = 1` 时才允许 `it_data_out_vld = 1`

所以顶层做了一层适配：

- 内核 `out_req` 仍直接接 `it_data_out_req`
- 官方口径下的 `it_data_out_vld` 被顶层额外与了一次 `it_data_out_req`

这样既保留了内部核的稳定保持行为，也满足了题面接口语义。

## 输出位宽假设

题面给的是：

- `it_data_out[39:0]`
- `4 x 10-bit`

而当前内部二维核输出仍是更高位宽中间结果。由于题面没有在现阶段资料里把最终规范化规则写死，这版先采用一个保守、可运行的打包假设：

- 单 lane 输出按有符号 `10-bit` 做饱和
- 饱和范围：`[-512, 511]`

对应函数在顶层里是 `pack_lane10()`。

这不是最终赛题定稿位宽策略，只是第一版官方接口打通用的临时规范化规则。后面拿到更完整的题面或参考实现后，应替换成最终规则。

## 当前价值

这版最大的价值不是“性能”，而是把下面三件事同时钉死：

1. 题面接口字段解码
2. sparse 输入按地址恢复到内部 TU
3. 内部高位宽结果如何适配到官方 `40-bit` 输出总线

后面继续扩模式、扩块型、补最终位宽规则时，都可以在这个顶层上继续推进，而不是再从子模块拼起。
